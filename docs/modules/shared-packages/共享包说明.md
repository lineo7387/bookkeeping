# 共享包说明

## 功能目标

共享包用于沉淀跨应用复用的类型、接口契约和请求能力，避免后台 Web、移动端、NestJS API 在字段命名和业务边界上重复定义。M0 阶段只建立最小可构建骨架，为后续真实 API 对接和业务模块开发预留稳定入口。

## 包边界

- `@bookkeeping/shared-types` 只放跨端共享的 TypeScript 类型、枚举和轻量接口契约，不包含运行时业务逻辑、不访问网络、不依赖前端或后端框架。
- `@bookkeeping/api-client` 只封装前端访问 NestJS 对外 API 的请求能力，不直接访问 FastAPI，不实现页面状态管理，不替代 NestJS 的认证、权限、事务和 AI 编排逻辑。
- 业务模块的复杂 DTO、校验规则和权限策略后续按模块逐步补充，避免在 M0 阶段提前扩张。

## shared-types 内容

当前 `@bookkeeping/shared-types` 提供以下基础类型：

- `ApiResponse`、`ApiError`、`ApiErrorCode`：统一 API 响应和错误码。
- `LedgerRole`、`LedgerPermission`、`LedgerSummary`：账本成员角色、能力和账本摘要信息。权限概念对应后端 `ledger_members` 与 Policy 层。
- `AccountType`、`AccountVisibility`、`AccountSummary`：账户类型、账户可见性和账户摘要信息。账户必须归属于 `ledgerId`，私密账户通过 `visibility` 与 `ownerId` 表达可见范围。
- `CategoryType`、`CategorySummary`：收入/支出分类类型和分类摘要信息。分类必须归属于 `ledgerId`，支持父级分类、系统分类标记、图标、颜色和排序字段。
- `TransactionType`、`TransactionVisibility`、`TransactionSource`、`TransactionSummary`：流水类型、可见性、来源和正式流水摘要信息。所有交易相关类型必须包含 `ledgerId`，不能建模为只归属于用户；AI 文本、票据识别和导入来源只表示正式流水的创建来源，不能绕过 NestJS 的确认与权限流程。
- `MonthlyStatisticsSummary`、`CategoryStatisticsSummary`、`AccountBalanceStatisticsSummary`、`MemberExpenseStatisticsSummary`：基础统计 API 的响应契约。金额和百分比均使用字符串，统计结果必须由 NestJS 根据当前用户权限过滤后返回。
- `AiTaskStatus`、`AiExtractionStatus`、`AiCandidateTransaction`、`AiExtractionSummary`：AI 任务、候选结果和候选状态。AI 候选结果只代表待确认建议，用户确认后才由 NestJS 创建正式交易。

金额字段统一使用字符串，例如 `"86.00"`；时间字段统一使用 ISO 8601 字符串，例如 `"2026-05-17T12:00:00.000Z"`，避免前端和后端出现浮点误差或时区歧义。

## api-client 职责

`@bookkeeping/api-client` 提供轻量 `BookkeepingApiClient` 和 `createApiClient` 工厂，封装：

- `POST /ledgers/:ledgerId/ai/text-parse`
- `GET /ai/tasks/:taskId`
- `POST /ai/extractions/:extractionId/confirm`
- `POST /ai/extractions/:extractionId/reject`

请求帮助函数返回 `ApiResponse<T>`，由调用方根据 `success`、`data`、`error` 做界面处理。网络请求失败、响应体读取失败、空响应、非标准响应结构等异常会被规范化为 `success: false` 的 `ApiResponse`，避免前端调用方遗漏 Promise rejection 分支。客户端只负责请求拼装、JSON 序列化和响应解析，不负责业务确认规则。

## NestJS-only 调用规则

所有前端应用只能通过 `@bookkeeping/api-client` 访问 NestJS 对外 API。`baseUrl` 必须配置为 NestJS API 地址，例如 `/api` 或 `https://api.example.com`。

禁止在 `@bookkeeping/api-client` 中配置、暴露或拼接 AI 内部服务地址。AI 文本解析、票据识别、候选保存、确认生成正式流水等流程都必须由 NestJS API 负责认证、账本成员权限校验、Policy 判断、任务状态维护和正式交易创建。

## 版本与构建验证

两个包均使用 ESM 友好的 `exports` 配置，构建产物输出到 `dist`，并生成 `.d.ts` 声明文件。每次修改共享包后至少运行：

```bash
pnpm --filter @bookkeeping/shared-types build
pnpm --filter @bookkeeping/api-client build
git diff --check
```

如果新增或调整 workspace 包依赖，应运行 `pnpm install` 更新锁文件。

## 后续扩展点

- 补充认证、账本、成员、账户、分类、流水、统计等模块 DTO。
- 增加分页、排序、筛选和上传相关的共享类型。
- 在 `api-client` 中按模块拆分资源客户端，但仍保持只调用 NestJS API。
- 增加共享校验包时，将可复用字段规则放入独立 `packages/validation`，避免类型包承担运行时校验职责。
- 根据后续多端需要增加错误码映射、请求追踪 ID、刷新 token 钩子和统一重试策略。
