# 后续脚手架计划

## 目标

本文用于交接后续脚手架与模块扩展准备工作。当前阶段记录执行顺序、用户需要运行的命令、Codex 创建后检查项和验收标准；Codex 不直接执行框架脚手架命令，也不提前创建未由用户确认的项目文件。

## 当前状态

已完成：

- 根目录 pnpm workspace 已初始化。
- `apps/api` 已创建，技术栈为 NestJS 11、Prisma 7、`@nestjs/config`。
- `apps/admin-web` 已创建，技术栈为 Vue 3、TypeScript、Vite、Tailwind CSS、`lucide-vue-next`。
- `apps/ai-service` 已由用户使用 `uv` 创建，技术栈为 FastAPI、Pydantic、pytest，当前已包含 M4 文本记账 internal-only 确定性 MVP parser。
- `packages/shared-types` 已创建，提供跨端共享类型与 AI 候选结果基础类型。
- `packages/api-client` 已创建，只封装前端访问 NestJS API 的请求能力。
- AI 服务规范、共享包说明、开发交接说明和脚手架协作协议已有中文文档。

尚未完成：

- `apps/mobile` 尚未创建。后续应采用 uni-app + Vue 3 + TypeScript。
- `packages/validation`、`packages/config` 尚未创建。建议等 API、移动端和共享类型稳定后再拆出。
- 当前分支与远端推送策略尚需用户确认，Codex 不应自行 push。

## 推荐顺序

1. 先确认当前 `codex/m4-ai-text-accounting` 的后续提交、合并或推送策略。
2. 如继续 M4，优先做 NestJS 与 FastAPI 的真实端到端联调、低置信度候选补全、任务筛选或后台展示增强。
3. 创建 uni-app `apps/mobile`，补齐多端用户端最小页面、请求配置和设计 token 对接。
4. 后续再评估 `packages/validation` 与 `packages/config`，避免过早抽象。

## GitHub 推送建议

推送必须由用户明确批准后再执行。未获得批准时，只能把以下命令作为建议，不得由 Codex 直接运行：

```bash
git status --short --branch
git log --oneline -8
git remote -v
```

如用户确认推送当前分支：

```bash
git push -u origin codex/m4-ai-text-accounting
```

如用户确认推送 `main`，应先确认当前分支、合并方式和远端保护规则，再由用户决定是否执行：

```bash
git checkout main
git pull --ff-only origin main
git merge --ff-only codex/m4-ai-text-accounting
git push origin main
```

验收标准：

- 用户明确确认要推送的分支和目标远端。
- `git status --short --branch` 可解释当前分支与未提交变更。
- 不在有未确认变更或未知冲突时推送。

## FastAPI AI 服务后续检查

### 用户执行命令建议

`apps/ai-service` 已由用户使用 `uv` 创建。以下 venv + pip 命令仅作为历史 fallback 记录，不是当前推荐继续执行的步骤：

```bash
mkdir -p apps/ai-service/app
cd apps/ai-service
python3 -m venv .venv
source .venv/bin/activate
python -m pip install --upgrade pip
python -m pip install fastapi "uvicorn[standard]" pydantic pydantic-settings httpx pytest
python -m pip freeze > requirements.txt
```

如果用户选择 `uv`、Poetry、Rye 或其他 Python 项目工具，应先确认工具方案和命令，再执行对应创建流程。

### Codex 后续检查

继续修改 AI 服务前，Codex 应检查：

```bash
test -d apps/ai-service
test -f apps/ai-service/pyproject.toml
test -f apps/ai-service/uv.lock
rg -n "fastapi|uvicorn|pydantic|pytest" apps/ai-service/pyproject.toml
```

随后应继续检查：

```bash
rg -n "FastAPI|/internal/ai/text-transaction|/internal/ai/receipt-ocr" apps/ai-service docs/modules/ai-service/AI服务规范.md
rg -n "候选结果|用户确认|NestJS|FastAPI" docs/modules/ai-service/AI服务规范.md .codex/ai-service-guidelines.md
```

### 验收标准

- `apps/ai-service` 只作为 internal-only FastAPI 服务存在。
- 前端、后台 Web、移动端都不能直接调用 FastAPI，只能调用 NestJS。
- 文本记账和票据识别只返回候选结果；候选结果必须由 NestJS 保存，用户确认后才创建正式流水。
- FastAPI 不处理登录态、账本成员权限、正式流水创建和主业务数据库写入。
- 日志不输出完整票据 URL、token、手机号、邮箱、API Key 或完整 OCR 原文。

## uni-app 移动端脚手架

### 用户执行命令建议

`apps/mobile` 应使用 uni-app + Vue 3 + TypeScript。uni-app 的创建方式取决于用户选择 HBuilderX、官方 CLI、Vite 插件方案或现有团队模板。执行前必须由用户确认具体 HBuilderX/CLI 流程和命令，Codex 不应凭空运行未知脚手架命令。

待用户确认后，可记录为类似形式：

```bash
# 示例说明：具体命令需按用户选择的 HBuilderX/CLI 流程确认后再执行
# create uni-app project at apps/mobile with Vue 3 + TypeScript
```

### Codex 创建后检查

用户执行命令后，Codex 应检查：

```bash
test -d apps/mobile
rg -n "uni-app|Vue|TypeScript|vite|pages" apps/mobile package.json pnpm-workspace.yaml
rg -n "NestJS|api-client|FastAPI" docs/modules/shared-packages/共享包说明.md .codex/development-rules.md
```

随后由 Codex 规范化时，应确认：

- `apps/mobile` 被纳入 pnpm workspace。
- 移动端请求层只使用 NestJS API 地址或 `@bookkeeping/api-client`，不拼接 FastAPI 内部地址。
- 页面和模块说明使用简体中文。
- 前端视觉优先遵循 `designer.md`，移动端可以更完整表达 High-Fidelity Claymorphism。
- AI 文本记账、票据识别页面只展示候选结果确认流程，不直接创建正式流水。

### 验收标准

- `apps/mobile` 可安装依赖并运行用户选择平台的基础开发命令。
- TypeScript、路径别名、环境变量和请求基地址有清晰约定。
- 至少有中文 README 或模块说明，说明移动端只调用 NestJS。
- 不存在前端直接调用 FastAPI 的配置、常量或示例代码。

## 后续共享包

`packages/validation` 和 `packages/config` 建议放在 FastAPI 与移动端之后处理。

建议触发条件：

- 多个端重复维护相同 DTO 校验规则时，再创建 `packages/validation`。
- 多个 TypeScript 包出现重复 tsconfig、eslint、prettier 配置时，再创建 `packages/config`。

验收标准：

- 新包必须有简体中文模块说明。
- 包边界清晰，不把业务流程塞进共享包。
- `shared-types` 继续只保存类型和轻量契约。
- `api-client` 继续只访问 NestJS，不直接访问 FastAPI。
